Реализация такой системы в коде --- это не один монолитный алгоритм, а
**набор согласованных модулей**, которые взаимодействуют по чёткому
потоку данных. Ниже я разберу архитектуру, структуру данных, логику
принятия решений и механизмы адаптации так, как это проектируется в
современных мобильных приложениях (без синтаксиса, только концепции и
паттерны).

**🏗️ 1. Архитектурная структура приложения**

Система строится по принципу **Offline-First + Event-Driven**:

- **Локальная БД** (SQLite/Realm/WatermelonDB) хранит всё: растения,
  логи, профили видов, калибровочные коэффициенты.

- **Движок расчётов** работает полностью на устройстве. Никаких запросов
  к серверу для принятия решений.

- **Фоновый планировщик** (WorkManager / BackgroundTasks) периодически
  пересчитывает состояние или реагирует на события (запуск приложения,
  изменение времени суток, ввод данных).

- **Реактивный UI** подписывается на изменения состояния растений и
  обновляет карточки/уведомления без перезагрузки.

**🔄 3. Машина состояний растения (Phase State Machine)**

Растение не находится в абстрактном «режиме ухода». Оно всегда в одной
из фаз: Dormant → Transition → ActiveGrowth → Flowering → PreDormant →
Dormant

**Как реализуется:**

- Каждая фаза имеет **входные/выходные условия** (пороговые значения
  температуры, длины дня, времени без полива, признаков роста).

- При запуске приложения или получении новых данных движок проверяет
  условия перехода.

- Переход меняет:

  - Базовый интервал высыхания

  - Допустимые действия (например, в Dormant блокируется water,
    разрешается только inspect)

  - Множители для расчёта индекса готовности

- Реализуется как конечный автомат (Finite State Machine) с таблицей
  переходов или как набор функций-предикатов, возвращающих next_phase.

**📐 4. Расчёт «Индекса готовности к уходу» (Readiness Engine)**

Это ядро системы. Не календарь, а **детерминированная функция**,
возвращающая число 0--100%.

**Формула концептуально:**

1.  Readiness =

2.  BaseIntervalProgress ×

3.  TemperatureFactor ×

4.  SubstrateFactor ×

5.  LightFactor ×

6.  PhaseMultiplier ×

7.  CalibrationOffset

Где:

- BaseIntervalProgress = сколько дней прошло с последнего полива /
  сколько дней в среднем сохнет субстрат этого вида

- TemperatureFactor = модель испарения. Выше t° → быстрее сохнет.
  Учитывает разницу день/ночь.

- SubstrateFactor = 80% пемзы сохнет в 2 раза быстрее, чем чистый торф.
  Задаётся коэффициентом влагоёмкости.

- LightFactor = фотосинтетически активный свет ускоряет метаболизм и
  транспирацию.

- PhaseMultiplier = 0.0 в покое, 0.3 в переходе, 1.0 в активном росте,
  1.2 при бутонизации.

- CalibrationOffset = динамический коэффициент, который система
  подтягивает на основе истории (см. ниже).

**Реализация в коде:**

- Функция вызывается при запуске приложения, при вводе новых данных или
  по фоновому таймеру.

- Возвращает объект: { index: 78, status: \"check_soon\", reason_chain:
  \[\"temp +24°C\", \"substrate 70% mineral\", \"phase active\",
  \"calibrated -2 days\"\] }

- UI рендерит статус, а не просто число.

**🧠 5. Механизм адаптации и калибровки (Feedback Loop)**

Система «учится» не через нейросети, а через **статистическую подстройку
параметров**. Это прозрачно, предсказуемо и работает офлайн.

**Алгоритм:**

1.  Пользователь вводит реальное состояние: soil_status: \"dry 2/3\",
    watered: true, volume: 30ml.

2.  Система сравнивает:

    - predicted_dry_days (по модели) vs actual_dry_days (по логу)

    - Вычисляет error_delta = actual - predicted

3.  Корректирует calibration_offset через экспоненциальное скользящее
    среднее (EMA):

> 1 new_offset = old_offset + α × error_delta\
> \
> Где α (коэффициент обучения) = 0.2--0.3. Чем больше данных, тем
> стабильнее коэффициент.

4.  Сохраняет пару predicted/actual в CalibrationHistory для отладки и
    экспорта.

5.  Следующий расчёт уже использует обновлённый коэффициент → прогноз
    становится точнее под **ваши** горшки, субстрат и климат.

**Почему не ИИ/ML?**

- Прозрачность: коллекционер видит, почему система сдвинула интервал.

- Надёжность: работает без интернета, не требует обучения на сервере, не
  выдаёт «чёрный ящик».

- Достаточность: физиология кактуса в теплице описывается
  детерминированными факторами. Статистическая калибровка покрывает 90%
  вариаций.

**⏱️ 6. Планировщик и уведомления (Scheduler Engine)**

Не cron-задачи с фиксированными датами, а **очередь событий с
приоритетами**.

**Как работает:**

1.  При пересчёте система проверяет readiness_index для всех активных
    растений.

2.  Если index \> threshold (например, 70%), создаётся задача:

    - type: \"check_moisture\" или type: \"water_recommended\"

    - priority: high/medium/low (зависит от фазы, риска, истории)

    - context: цепочка причин, рекомендуемый объём, метод полива

3.  Задача попадает в локальную очередь уведомлений.

4.  Фоновый сервис (WorkManager / BGTask) доставляет
    push/local-уведомление в заданное время окна (например, 8:00--10:00,
    когда удобно проверять теплицу).

5.  При открытии приложения пользователь видит задачи, вводит результат
    → лог обновляется → очередь пересчитывается.

**Защита от спама:**

- Группировка по локациям

- Quiet hours

- Дебаунс: если пользователь уже открыл карточку и ввёл данные,
  дублирующее уведомление отменяется.

**🔗 7. Полный поток данных (Data Flow)**

\[Старт приложения / Изменение времени / Ввод данных\]

↓

\[Загрузка PlantInstance + SpeciesTemplate + последние CareLogs\]

↓

\[PhaseEngine определяет текущую фазу\]

↓

\[ReadinessEngine считает индекс готовности\]

↓

\[Scheduler генерирует/обновляет задачи\]

↓

\[UI обновляет карточки, календарь, уведомления\]

↓

\[Пользователь вводит результат проверки/полива\]

↓

\[CareLog сохраняется → CalibrationHistory обновляется\]

↓

\[EMA-коррекция коэффициентов → следующий цикл точнее\]

### 🛠️ 8. Технические нюансы реализации

  ----------------------------------------------------------------------
           **Аспект**          **Как решается**
  ---------------------------- -----------------------------------------
     \*\*Офлайн-работа\*\*     Все расчёты локально. БД в SQLite/Room.
                               Нет обязательных API-запросов.

   \*\*Производительность\*\*  Расчёт индекса только при изменении
                               входных данных или по таймеру (не в

                               результатов.

**💡 Итог: как это выглядит в коде архитектурно**

1.  **Data Layer**: локальная БД с таблицами растений, видов, логов,
    среды, калибровки.

2.  **Logic Layer**:

    - PhaseStateMachine (переходы по порогам)

    - ReadinessCalculator (детерминированная формула с весами)

    - CalibrationEngine (EMA-подстройка на основе истории)

3.  **Scheduler Layer**: очередь задач, приоритизация, доставка
    уведомлений, quiet hours.

4.  **UI Layer**: реактивные подписки на изменения состояния, рендер
    статуса + контекста, формы ввода логов.

5.  **Sync/Backup Layer** (опционально): шифрованный экспорт/импорт
    JSON, облачная синхронизация по требованию.

Система не «гадает», а **моделирует физиологию + среду + историю**,
выдавая прозрачные, калибруемые рекомендации. Она работает как цифровой
лаборант: фиксирует, считает, предлагает, запоминает ошибки, но
оставляет последнее слово за коллекционером. Именно так и строится
надёжное адаптивное приложение для живой коллекции.
