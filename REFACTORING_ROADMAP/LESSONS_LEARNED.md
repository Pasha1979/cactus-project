## Как использовать этот файл

### При возникновении ошибки:
1. Записать дату и контекст (какой шаг рефакторинга выполнялся)
2. Описать ошибку (что случилось)
3. Описать причину (почему это случилось)
4. Описать исправление (как я это исправил)
5. Описать урок (чему я научился)
6. Классифицировать ошибку по типу
7. **КРИТИЧЕСКИЙ ШАГ:** Определить, какие проверки добавить, чтобы избежать этой ошибки в будущем
8. **КРИТИЧЕСКИЙ ШАГ:** Определить, какие изменения в процесс работы нужны, чтобы предотвратить повторение

### Периодически:
1. **ПЕРЕД КАЖДЫМ ШАГОМ** рефакторинга просматривать этот файл
2. Проверять, не повторяю ли я ту же ошибку
3. Обновлять уроки по мере накопления опыта
4. Проверять, работают ли добавленные проверки
5. Если ошибка повторяется - это критический провал системы обучения, нужно пересмотреть процесс

### КРИТИЧЕСКОЕ ПРАВИЛО:
**НИ ОДНА ОШИБКА НЕ ДОЛЖНА БЫТЬ ПОТЕРЯНА**
- ВСЕ ошибки без исключения должны быть записаны в этот файл
- Каждая ошибка должна быть проанализирована глубоко
- Каждая ошибка должна привести к улучшению процесса
- Система обучения должна работать постоянно

### Ошибка 5: Неполная интеграция исключений — добавлены exceptions, но не failures и не handler

**Дата:** 2026-05-10
**Категория:** Архитектурная
**Контекст:** Аудит шагов 1.1-1.7, задача 1.7.13

**Описание ошибки:**
- Добавлены `DuplicateIdException` и `OAuth2Exception` в exceptions.dart
- НЕ добавлены `DuplicateFailure` и `OAuth2Failure` в failures.dart
- НЕ обновлен `ErrorHandler.handleException()` для новых исключений
- Результат: новые исключения будут отдавать `UnknownFailure` вместо специфичных ошибок

**Корневая причина:**
1. **Непонимание архитектуры слоёв:** Exception (data слой) → Failure (domain слой) → UI. Добавили только data-слой.
2. **Отсутствие системного подхода:** не было чек-листа "что менять при добавлении нового исключения"
3. **Фрагментарное выполнение:** flutter analyze прошел → "всё хорошо", хотя логика неполная
4. **Отсутствие понимания Clean Architecture:** не понимал что исключения data-слоя должны преобразовываться в failures domain-слоя

**Исправление:**
- Добавить DuplicateFailure и OAuth2Failure в failures.dart
- Обновить ErrorHandler.handleException()
- Создать системный подход: при добавлении исключения — менять 3 файла

**Урок:**
- **ПРАВИЛО ТРЁХ ФАЙЛОВ:** новое исключение требует изменений ВСЕГДА в 3 файлах: exceptions.dart → failures.dart → error_handler.dart
- **EXCEPTION ≠ FAILURE:** Exception — data слой, Failure — domain слой. Нельзя добавлять одно без другого.
- **АРХИТЕКТУРНЫЙ ЧЕК-ЛИСТ:** при изменении архитектурного компонента проверять весь pipeline

**Связанные шаги:**
- Шаг 1.2: core/error, Шаг 1.7: Repository Pattern
- Файлы: exceptions.dart, failures.dart, error_handler.dart

### Ошибка 7: HiveDatabase static без защиты от повторной инициализации

**Дата:** 2026-05-10
**Категория:** Архитектурная
**Контекст:** Аудит шагов 1.1-1.7, задача 1.6.6-1.6.7

**Описание ошибки:**
- `HiveDatabase.initialize()` использует static поля
- `Hive.registerAdapter()` выбросит исключение при повторном вызове
- Нет флага `_isInitialized` для защиты
- Невозможен hot-restart без crash
- Static поля нельзя мокировать в тестах

**Корневая причина:**
1. **Недостаточное знание API Hive:** не знал что registerAdapter выбрасывает при повторном вызове
2. **Отсутствие edge-case анализа:** не рассматривал сценарии hot-restart, тестирования
3. **Спешка:** быстрая реализация "чтобы работало" без продумывания жизненного цикла
4. **Отсутствие тестов:** если бы написал тест `testInitializeTwice`, ошибка была бы найдена

**Исправление:**
- Добавить `_isInitialized` флаг
- Добавить `_isInitializing` флаг для предотвращения race condition
- Оборачивать `registerAdapter` в try-catch на случай повторной регистрации

**Урок:**
- **EDGE-CASE АНАЛИЗ:** при проектировании singleton/static всегда рассматривать: повторный вызов, concurrent вызов, тестирование
- **API ЗНАНИЕ:** перед использованием стороннего API проверять edge cases в документации
- **HOT-RESTART ДЛЯ FLUTTER:** любой static singleton ДОЛЖЕН поддерживать hot-restart
- **TESTABILITY:** static поля убивают тестируемость. Использовать dependency injection.

### Ошибка 10: Опасные No-op заглушки для CRUD операций (2026-05-10)

**Дата:** 2026-05-10
**Категория:** Безопасность заглушек

**Описание ошибки:**
- При создании реализаций репозиториев (И.3) оставлены 4 No-op заглушки для CRUD операций:
  - `saveGlobalWateringDates()` — пустое тело, данные не сохраняются
  - `syncWithCloud()` — пустое тело, синхронизация не происходит
  - `createBatch()` — пустое тело, партия не создаётся
  - `updateBatch()` — пустое тело, обновление не сохраняется
- No-op заглушки для CRUD — КРИТИЧЕСКИ опасны: пользователь думает что операция выполнилась, но данные потеряны молчаливо

**Причина ошибки:**
- Не было системы классификации заглушек по уровню риска
- Не было правила "No-op запрещён для CRUD"
- Не было реестра заглушек для контроля
- Использовалась привычка "поставить TODO и пустой return" вместо безопасной заглушки

**Исправление:**
- Создана система управления заглушками в CHECKLIST.md:
  - Классификация по типам (No-op, Return empty, Hardcoded, Partial impl)
  - Уровни риска (Критический/Средний/Низкий)
  - Алгоритм создания безопасной заглушки
  - Таблица "Безопасная vs Опасная заглушка" для каждой операции
- Все 4 опасные No-op заглушки заменены на `throw UnimplementedError()`:
  - `saveGlobalWateringDates()` → throw с описанием
  - `syncWithCloud()` → throw с описанием
  - `createBatch()` → throw с описанием
  - `updateBatch()` → throw с описанием
- Добавлено правило 16 в PROMPT.md: "УПРАВЛЕНИЕ ЗАГЛУШКАМИ"
- Создан полный реестр 9 заглушек с анализом рисков

**Урок:**
- No-op заглушки для CRUD — ЗАПРЕЩЕНЫ. Вместо них использовать `throw UnimplementedError()`
- `throw UnimplementedError()` предпочтительнее No-op: ошибка видна сразу, а не молчаливо
- Каждая заглушка должна быть занесена в реестр с анализом риска
- Перед созданием заглушки — определить тип и выбрать безопасный вариант

**Связанные шаги:**
- Фаза И.3: Создание реализаций репозиториев
- Шаг 1.10.10: WateringProvider (заглушка saveGlobalWateringDates)
- Шаг 1.10.11: BatchProvider (заглушки createBatch, updateBatch)
- Шаг 1.11.15: SyncManager (заглушка syncWithCloud)

**Проверки для предотвращения:**
- При создании заглушки: определить тип по алгоритму CHECKLIST.md
- Если No-op для CRUD: заменить на `throw UnimplementedError()`
- После создания заглушки: занести в реестр CHECKLIST.md
- Перед шагом с заглушками: реализовать или подтвердить безопасность

### 2026-05-10: Dependency Injection — выбор подхода (1.9)

**Описание:**
При реализации DI выбор между `injectable` (кодогенерация) и ручной регистрацией `get_it`.

**Анализ проведён:**

1. **Прогноз рисков (Правило 17):**
   - injectable + build_runner: требует аннотаций @singleton/@injectable на ВСЕХ классах
   - Проблема: репозитории-заглушки (throw UnimplementedError) — кодогенерация может сломаться
   - Проблема: DataSource'ы требуют Box в конструкторе — не все конструкторы без параметров
   - Проблема: build_runner требует запуска при каждом изменении — медленно
   - Решение: ручная регистрация в get_it — проще, контролируемо, без codegen

2. **Сохранение функциональности (Правило 18):**
   - Ручная регистрация идентична injectable по результату (get_it контейнер)
   - Отличие: нет кодогенерации, регистрация явная в одном файле
   - Преимущество: видны все зависимости, легко отлаживать

3. **Результат:**
   - Создан `injection_container.dart` с явной регистрацией 5 DataSources + NetworkInfo + 8 Repositories
   - get_it `sl<T>()` используется для получения зависимостей
   - Инициализация DI в main.dart после Hive, перед миграцией
   - flutter analyze проходит без issues

**Урок:**
- Не всегда кодогенерация лучше ручного подхода
- Ручная регистрация DI: проще, быстрее, легче отлаживать
- injectable имеет смысл при >20 зависимостях или сложной иерархии
- Важно документировать порядок инициализации (Hive → DI → Migration)

**Связанные шаги:**
- Шаг 1.9: Dependency Injection

### 2026-05-10: Разделение PlantProvider — Strangler Fig Pattern (1.10)

**Описание:**
PlantProvider — монолит 2478 строк, 13 зон ответственности. Полное разделение за одну сессию невозможно безопасно.

**Анализ проведён:**

1. **Прогноз рисков (Правило 17):**
   - Прямое удаление PlantProvider сломает 20+ экранов
   - Методы имеют скрытые зависимости (`_plants`, `_hasUnsavedChanges`, `notifyListeners` + `savePlants`)
   - Нужна стратегия "Strangler Fig" — постепенное вытеснение
   - Сначала создать новые провайдеры, оставить старый как фасад

2. **Сохранение функциональности (Правило 18):**
   - Старый PlantProvider остаётся на месте — все экраны продолжают работать
   - Новые провайдеры зарегистрированы в MultiProvider параллельно
   - Постепенно экраны переходят на новые провайдеры по одному

3. **Архитектура новых провайдеров:**
   - PlantCrudProvider: CRUD + selection + helpers
   - WateringProvider: global/individual watering + recommendations
   - WinteringProvider: dates + temperature + log entries
   - PhotoProvider: user/llifle/adult photos + sync flags
   - BatchProvider: batch/seedling logic
   - SyncProvider: backup + restore + JSON merge
   - QrCodeProvider: QR codes + files + scan history
   - WeatherProvider: location + weather + advice

4. **Результат:**
   - 8 новых провайдеров созданы в `presentation/providers/`
   - Интегрированы в MultiProvider (Android + Windows)
   - flutter analyze: 0 errors, только info `prefer_final_fields`
   - Старый PlantProvider сохранён — приложение работает

**Урок:**
- Не пытаться разделить монолит за один раз — это гарантированно сломает приложение
- Strangler Fig Pattern: создать новое → интегрировать параллельно → постепенно мигрировать → удалить старое
- Каждый новый провайдер должен быть автономным (свои данные + свои notifyListeners)
- `barrel file` (`providers.dart`) упрощает импорты
- Интеграция MultiProvider — добавить ChangeNotifierProvider для каждого нового

## Ошибка: Потеря критической функциональности облачной синхронизации при рефакторинге PlantProvider → PlantCrudProvider

**Дата:** 2026-05-11
**Контекст:** Шаг 1.10 — разделение PlantProvider на специализированные провайдеры. Замена всех ссылок PlantProvider на PlantCrudProvider в cloud_storage_provider, welcome_screen, plant_cards, notes_bottom_sheet, main.dart. Удаление старого PlantProvider.

### Что случилось
При рефакторинге `PlantCrudProvider.loadFromCloudJson()` была написана упрощённая реализация, которая просто заменяла локальные растения облачными:
```dart
_plants = plantsJson.map(...).toList(); // ❌ Потеря локальных данных!
```

Аналогично `toJson()` экспортировал только `plants` и `lastLocalUpdate`, пропуская:
- `globalWateringDates`
- `adultImages`
- `winteringStartDate/EndDate/Temperature`
- `winteringLogEntries`

### Почему это случилось
1. **Фокус на flutter analyze вместо функциональности.** Проверял только отсутствие ошибок компиляции, не анализировал логику методов.
2. **Недостаточное сравнение с оригиналом.** Не сделал `git diff` старого `PlantProvider.loadFromCloudJson` и `toJson` перед удалением файла.
3. **Предположение, что "заглушки достаточно".** Считал, что TODO-заглушки для `ensureLocalPhotosExist`, `cleanupUnusedPhotosForSelected` и т.д. временно приемлемы, но не заметил, что критический merge-алгоритм тоже стал заглушкой.
4. **Отсутствие финальной проверки "до vs после".** Не провёл сравнение списка методов и их реализаций между старым и новым провайдером.

### Последствия
- **Облачная синхронизация между устройствами сломана:** локальные растения, которых нет в облаке, удалялись при загрузке.
- **Данные других провайдеров (WateringProvider, WinteringProvider) терялись в облаке** при upload, потому что `toJson` их не сериализовал.
- **Пользователь бы потерял данные** при первой же синхронизации после обновления.

### Как исправлено
1. Восстановлена полная логика `loadFromCloudJson` из старого `PlantProvider`:
   - Merge растений по `permanentId` с выбором более свежей версии по `lastModified`
   - Сохранение локальных растений, отсутствующих в облаке
   - Слияние фото с дедупликацией и учётом удалённых
   - Слияние заметок, цветения, проращивания, поливов
2. Расширен `toJson` для включения всех legacy-полей (`globalWateringDates`, `adultImages`, `wintering*`).
3. Добавлены legacy-поля в `PlantCrudProvider` для обратной совместимости формата облачного JSON.
4. Убран дублирующий `notifyListeners()` из `cloud_storage_provider` (вызывался и в `loadFromCloudJson`, и явно после).
5. Скопировано в Windows.

### Урок
> **Ни один метод не может быть "упрощён" без явного сравнения с оригиналом.**
> 
> При миграции провайдера:
> 1. **До удаления старого файла** — сделать `git show HEAD:old_file > temp.txt` и сравнить каждый метод
> 2. **Проверить не только сигнатуру, но и тело метода**
> 3. **Особое внимание к методам синхронизации, сериализации, merge-логике**
> 4. **После рефакторинга — финальная проверка:** `diff <(git show HEAD:old_provider | grep -E "^\s+\w+.*\(") <(cat new_provider | grep -E "^\s+\w+.*\(")`

**Проверки, которые нужно добавить в процесс:**

> 1. При старте приложения — данные загружаются с диска?
> 2. При изменении через UI — данные сохраняются на диск?
> 3. После перезапуска приложения — данные восстановились?
> 4. Если провайдер использует SharedPreferences — есть ли `load()` и `_save()`?
> 5. Если провайдер получает данные из облака — сохраняет ли он их локально?
