## Как использовать этот файл

### При возникновении ошибки:
1. Записать дату и контекст (какой шаг рефакторинга выполнялся)
2. Описать ошибку (что случилось)
3. Описать причину (почему это случилось)
4. Описать исправление (как я это исправил)
5. Описать урок (чему я научился)
6. Классифицировать ошибку по типу
7. **КРИТИЧЕСКИЙ ШАГ:** Определить, какие проверки добавить, чтобы избежать этой ошибки в будущем
8. **КРИТИЧЕСКИЙ ШАГ:** Определить, какие изменения в процесс работы нужны, чтобы предотвратить повторение

### При УСПЕШНОМ решении (не только ошибки!):
1. Что сделано хорошо — записать как паттерн
2. Почему это сработало — понять причину
3. Можно ли применить к другим шагам — обобщить
4. Добавить в раздел «Успешные паттерны» ниже

### Периодически:
1. **ПЕРЕД КАЖДЫМ ШАГОМ** рефакторинга просматривать этот файл
2. Проверять, не повторяю ли я ту же ошибку
3. Обновлять уроки по мере накопления опыта
4. Проверять, работают ли добавленные проверки
5. Если ошибка повторяется - это критический провал системы обучения, нужно пересмотреть процесс

### КРИТИЧЕСКОЕ ПРАВИЛО:
**НИ ОДНА ОШИБКА НЕ ДОЛЖНА БЫТЬ ПОТЕРЯНА**
- ВСЕ ошибки без исключения должны быть записаны в этот файл
- Каждая ошибка должна быть проанализирована глубоко
- Каждая ошибка должна привести к улучшению процесса
- Система обучения должна работать постоянно

### Ошибка 4: КРИТИЧЕСКАЯ — Опечатка в маппинге полей PlantRepositoryImpl (потеря данных)

**Дата:** 2026-05-10
**Категория:** Логическая / Миграционная
**Контекст:** Аудит шагов 1.1-1.7 после завершения Шага 1.7 (Repository Pattern)

**Описание ошибки:**

- Пользователь не получит напоминания об удобрении

**Корневая причина (глубокий анализ):**

3. **Отсутствие unit-тестов:** при наличии теста `testMapToDtoPreservesAllFields()` ошибка была бы обнаружена немедленно
4. **Спешка:** при быстрой реализации не было пошаговой верификации каждого поля
5. **Отсутствие типобезопасного маппинга:** использование JSON как промежуточного формата вместо прямого присвоения полей сделало код хрупким

**Исправление:**
- Исправить на `plannedFertilizationDate: entity.plannedFertilizationDate`
- Проверить ВСЕ поля в маппинге попарно (entity.field → dto.field)
- Заменить JSON-посредника на прямое присвоение полей

**Урок:**
- **РУЧНОЙ МАППИНГ БОЛЕЕ 20 ПОЛЕЙ — КРИТИЧЕСКИЙ РИСК:** при маппинге >10 полей обязательна верификация через чек-лист

- **UNIT-ТЕСТЫ ДЛЯ МАППИНГА:** обязательны тесты на сохранение всех полей при round-trip (entity → dto → entity)

**Связанные шаги:**

### Ошибка 5: Неполная интеграция исключений — добавлены exceptions, но не failures и не handler

**Дата:** 2026-05-10
**Категория:** Архитектурная
**Контекст:** Аудит шагов 1.1-1.7, задача 1.7.13

**Описание ошибки:**
- Добавлены `DuplicateIdException` и `OAuth2Exception` в exceptions.dart
- НЕ добавлены `DuplicateFailure` и `OAuth2Failure` в failures.dart
- НЕ обновлен `ErrorHandler.handleException()` для новых исключений
- Результат: новые исключения будут отдавать `UnknownFailure` вместо специфичных ошибок

**Корневая причина:**
1. **Непонимание архитектуры слоёв:** Exception (data слой) → Failure (domain слой) → UI. Добавили только data-слой.
2. **Отсутствие системного подхода:** не было чек-листа "что менять при добавлении нового исключения"
3. **Фрагментарное выполнение:** flutter analyze прошел → "всё хорошо", хотя логика неполная
4. **Отсутствие понимания Clean Architecture:** не понимал что исключения data-слоя должны преобразовываться в failures domain-слоя

**Исправление:**
- Добавить DuplicateFailure и OAuth2Failure в failures.dart
- Обновить ErrorHandler.handleException()
- Создать системный подход: при добавлении исключения — менять 3 файла

**Урок:**
- **ПРАВИЛО ТРЁХ ФАЙЛОВ:** новое исключение требует изменений ВСЕГДА в 3 файлах: exceptions.dart → failures.dart → error_handler.dart
- **EXCEPTION ≠ FAILURE:** Exception — data слой, Failure — domain слой. Нельзя добавлять одно без другого.

### 2026-05-10: Dependency Injection — выбор подхода (1.9)

- Не всегда кодогенерация лучше ручного подхода

### Ошибка 18: Отсутствие индексов Hive + неэффективный getPlantById (2026-05-11)

**Дата:** 2026-05-11
**Категория:** Производительность / Архитектурная
**Контекст:** Аудит шагов 1.1–1.12. Шаг 1.6 (Hive Database) и 1.7 (Repository Pattern).

**Описание ошибки:**
- `PlantLocalDataSource.getPlantById()` использовал `firstWhere` по `permanentId` вместо `box.get(id)`
- Это превращало O(1) операцию по ключу Hive в O(n) линейный поиск
- Для 1000+ растений это давало ~1000-кратное замедление поиска по ID
- `getPlantsByStatus` и `getPlantsByCategory` также использовали O(n) фильтрацию по всему box
- Индексы Hive не были созданы, хотя были запланированы в 1.6.11
- Причина отсрочки: «сейчас O(n) незаметен для небольших объёмов» — неверный подход для приложения, которое может быть передано другому пользователю с большой коллекцией

**Корневая причина:**
1. Недооценка масштабируемости: решение «работает сейчас» вместо «работает всегда»
2. Непонимание механики Hive: ключ box = permanentId, `box.get(id)` — нативный O(1)
3. Отсутствие индексов для вторичных ключей (status, category, displayId)
4. Пользователь (правильно) потребовал адаптировать приложение для 1000+ растений сразу

**Исправление:**
- `getPlantById` → `_plantBox.get(id)` (O(1))
- Создан `PlantIndexManager` — собственный слой индексов поверх Hive:
  - Ключ: `"field:value"`, значение: `List<String>` permanentIds
  - Индексы по `status`, `category`, `displayId`
  - Авто-обновление при `addPlant` / `updatePlant` / `deletePlant`
  - `rebuildIndex()` для миграций и восстановления
- `getPlantsByStatus` / `getPlantsByCategory` — O(1) получение списка id + O(k) загрузка объектов
- Fallback к O(n) оставлен для случаев без индекса
- `DataMigrationManager._migratePlants` — вызывает `rebuildIndex()` после миграции
- Файлы синхронизированы Android ↔ Windows

**Урок:**
- **Приложение должно быть адаптировано для масштабирования сразу**, а не «когда-нибудь»
- Не экономить на инфраструктуре данных — индексы стоят дешевле, чем переделка позже
- Знать особенности используемой БД: в Hive `box.get(key)` — O(1), `.values.firstWhere` — O(n)
- Вторичные индексы в Hive нужно реализовывать явно, но это несложно

**Проверки для предотвращения:**
- При реализации DataSource — проверять: используются ли нативные O(1) операции БД?
- `grep 'firstWhere.*permanentId'` — должен быть 0 в production коде
- При аудите: оценивать сложность операций (Big O) для каждого CRUD метода
- Если планируется отложить индексы/оптимизацию — обязательно добавить триггер условия (количество записей, время отклика)

### Ошибка 25: Ошибка миграции дат полива - type 'List<dynamic>' is not a subtype of type 'String?'

- Проводить тестирование миграции на реальных данных пользователя перед деплоем
- Добавлять fallback механизмы для обработки неожиданных форматов данных
- Всегда проверять старый код на предмет того, как данные сохранялись

### Ошибка 6: Deep link redirect блокировал внешние deep links
**Дата:** 2026-05-12
**Категория:** Рефакторинг (go_router)
**Контекст:** Шаг 2.1.5 — Deep linking

**Описание:**
- `redirect` в GoRouter перенаправлял ВСЕ запросы на `/welcome`, если пользователь не видел welcome screen
- Внешний deep link `cactusapp://plant/123` перенаправлялся на `/welcome` вместо открытия карточки растения

**Причина:**
- Не было проверки на то, является ли запрос deep link (путь != '/')
- Комментарий "Deep link: не редиректим внешние URL" был написан, но реализация отсутствовала

**Исправление:**
- Добавлено условие `if (state.uri.path != '/') return null;` в redirect (Android + Windows)
- Теперь редирект срабатывает только для корневого запуска приложения

**Урок:**
- Redirect должен различать initial launch и deep link
- Deep links должны всегда иметь приоритет над welcome redirect
- Тестировать deep links после реализации redirect
