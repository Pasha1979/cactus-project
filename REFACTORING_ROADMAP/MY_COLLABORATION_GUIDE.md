# Руководство по совместной работе (MY_COLLABORATION_GUIDE)

**Дата создания:** 2026-05-12
**Цель:** Адаптация процесса рефакторинга под стиль работы пользователя, минимизация ошибок, обучение на успехах.

## Профиль пользователя (НАБЛЮДАЕМАЯ ПЕРЕМЕННАЯ, не константа)

> **Критично:** Этот профиль — не догма. Я отслеживаю твои реакции и адаптируюсь.
> Ты можешь менять подход в зависимости от сложности, интересности или контекста.
> Я должен УВИДЕТЬ это изменение и подстроиться.

| Характеристика | Проявление | Моя реакция | Когда меняется |
|----------------|------------|-------------|--------------|
| **Консервативность** | Обычно предпочитает Вариант А (минимальный риск) | Начинать с Варианта А, но предлагать Вариант Б если интересно | Может выбрать Вариант Б если задача простая или предложение убедительное |
| **Глубокие аудиты** | Требует 9-10 пунктных проверок после каждого шага | Проводить полный аудит БЕЗ запроса — автоматически | Не меняется — железное правило |
| **Кроссплатформенность** | Android + Windows, платформенные различия критичны | Всегда параллельно, всегда проверять оба проекта | Не меняется — железное правило |
| **Синхронизация фото** | Долго настраивал, критично для него | Никогда не ломать без явного разрешения | Не меняется — железное правило |
| **Обучение** | Хочет чтобы я учился и совершал меньше ошибок | Записывать ВСЕ, анализировать, предсказывать | Может усилить требования после повторной ошибки |
| **Предостережение** | Хочет чтобы я его предостерегал от ошибок | Активно предупреждать о рисках, а не ждать | Может попросить «просто сделай» если устал от обсуждений |

## Железные правила (добавлены после 23 ошибок)

### 1. Правило «9 пунктов» — автоматический аудит
**ПОСЛЕ КАЖДОГО шага (даже микро-шага):**
1. Все ли выполнено в полном объеме, оба проекта, ничего не потеряно?
2. Все ли функции и фичи сохранены, рабочие?
3. Сохранены ли платформенные различия?
4. Нет скрытых багов для будущих шагов?
5. Нет скрытых багов для запуска приложений?
6. Код, файлы, связи, зависимости логичны и рабочие?
7. Нет TODO/FIXME/HACK/XXX/STUB?
8. `flutter analyze` чистый на ОБОИХ проектах?
9. Все соответствует целям рефакторинга?

> **Критично:** Не ждать запроса — провести автоматически и отчитаться.

### 2. Правило «Вариант А» — начальная точка, не догма
**ПЕРЕД любым изменением, которое может затронуть:**
- Синхронизацию фото (Яндекс Диск)
- Платформенную сборку (особенно Windows)
- Данные пользователя (Hive, SharedPreferences)

**Действие:**
1. Начать с Варианта А (минимальный риск) — это моя отправная точка
2. Предложить Вариант Б (полный) если он логичен и риски оправданы
3. Объяснить риски каждого честно, без преувеличения
4. Подождать выбор пользователя
5. **Никогда не выбирать за него**
6. **Если пользователь выбирает Вариант Б — зафиксировать это в профиле как «готов к риску в этом контексте»**

### 3. Правило «Предсказание ошибок» — до кода
**ПЕРЕД написанием/изменением кода:**
1. Проанализировать что может сломаться
2. Предсказать 3 самых вероятных ошибки
3. Спросить пользователя о рисках
4. Создать бэкап если изменение критичное

### 4. Правило «Инструменты — самопроверка»
**ПЕРЕД каждым вызовом инструмента:**
1. Все ли обязательные параметры заполнены?
2. Нет ли пустых `{}`?
3. Правильные ли типы данных?
4. Если сомнение — перечитать документацию

### 5. Правило «Платформа — проверка pubspec»
**ПЕРЕД копированием файла Android → Windows:**
1. Проверить `pubspec.yaml` Windows — есть ли все импортируемые пакеты?
2. Проверить platform-specific код в оригинальном Windows-файле
3. Использовать `git diff` для проверки

### 6. Правило «Безопасность парсинга»
**Любой новый `fromJson` или изменение парсинга:**
1. Использовать `DateTime.tryParse` вместо `DateTime.parse`
2. Добавлять fallback (`?? defaultValue`)
3. Проверить `grep DateTime.parse\(` по ВСЕМУ проекту
4. Проверить вложенные модели

## Как я отслеживаю изменения в твоём стиле

### Сигналы к адаптации:
| Сигнал | Что это значит | Моя реакция |
|--------|---------------|-------------|
| «Давай попробуем Вариант Б» | Готов к риску в этом контексте | Запомнить: для похожих задач предлагать оба варианта сразу |
| «Просто сделай, не объясняй» | Устал от обсуждений, хочет результат | Сократить преамбулу, действовать быстрее |
| «Это слишком рискованно» | Вернулся к консервативности | Усилить анализ рисков, предлагать только Вариант А |
| «Почему ты не предупредил?» | Хочет больше предостережений | Активнее предупреждать, не ждать |
| «Давай проверим всё снова» | Недоверие после ошибки | Провести более глубокий аудит |
| «Отлично, двигаемся дальше» | Доверие высокое, темп можно ускорить | Можно предлагать более смелые решения |

### Правило фиксации:
После каждой сессии обновлять этот файл если:
- Пользователь изменил типичное поведение
- Появился новый паттерн реакций
- Железное правило подтвердилось или опроверглось

## Успешные практики (что работает — записывать)

### Работает хорошо:
1. **Микро-шаги** — разбиение на маленькие задачи с коммитами
2. **Параллельные изменения** — Android + Windows одновременно
3. **`flutter analyze` после каждого шага** — автоматическая проверка
4. **LESSONS_LEARNED.md** — запись ошибок с анализом
5. **UiState pattern** — чистая архитектура, обработка ошибок
6. **Консервативный подход** — Вариант А как отправная точка для рисков

### Что усиливать:
1. **Предсказание ошибок** — пользователь хочет чтобы я предупреждал
2. **Анализ рисков** — до кода, а не после
3. **Проверка инструментов** — самопроверка перед каждым вызовом
4. **Проверка вложенных моделей** — при любых изменениях парсинга
5. **Адаптация в реальном времени** — следить за реакциями и меняться
