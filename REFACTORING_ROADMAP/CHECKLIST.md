## ЧЕК-ЛИСТ: УПРАВЛЕНИЕ TODO В КОДЕ (ПРАВИЛА ПРОГНОЗИРОВАНИЯ)

### При создании TODO — АНАЛИЗ ПЕРЕД ПАУЗОЙ

**Вопрос 1: Могу ли я реализовать это СЕЙЧАС?**
- [ ] Если ДА — реализовать немедленно, не оставлять TODO
- [ ] Если НЕТ — перейти к анализу зависимостей

**Вопрос 2: Почему я не могу реализовать сейчас?**
- [ ] Нужны другие компоненты, которые ещё не созданы? → Зависимость
- [ ] Нужна бизнес-логика, которую ещё не выделили? → Архитектурная зависимость
- [ ] Нужна миграция данных из старого хранилища? → Данные
- [ ] Нужно добавить поле в DTO? → Изменение модели
- [ ] Нужен сервис/провайдер, который создаётся на следующем шаге? → Слой

**Вопрос 3: Какая минимальная безопасная заглушка?**
- [ ] Функция возвращает пустой список (`return []`) — безопасно для UI
- [ ] Функция возвращает null (`return null`) — проверить, что caller обрабатывает null
- [ ] Функция ничего не делает (`return;`) — проверить, что это не ломает логику
- [ ] Заглушка не скрывает критическую ошибку (например, не возвращает фейковые данные как реальные)

**Вопрос 4: Где естественно реализовать этот TODO?**
- [ ] TODO с DTO/моделью → шаг, где работаем с этой моделью
- [ ] TODO с CRUD → шаг, где создаём соответствующий Provider
- [ ] TODO с API/синхронизацией → шаг сервисного слоя
- [ ] TODO с миграцией данных → шаг миграции
- [ ] TODO с UI-логикой → шаг разделения Provider'ов

### При планировании реализации TODO — ВЫБОР ШАГА

**Критерий 1: Минимизация зависимостей**
- [ ] Все необходимые компоненты для TODO уже существуют на момент шага? ИЛИ создаются в этом же шаге?
- [ ] TODO не требует компонентов из БУДУЩИХ шагов?

**Критерий 2: Естественность предметной области**
- [ ] TODO логически связан с целью шага?
- [ ] Реализация TODO не выглядит "привязанной" к чужому шагу?

**Критерий 3: Отсутствие блокировки**
- [ ] TODO не блокирует следующие шаги (можно продолжить без него)?
- [ ] Если блокирует — он ДОЛЖЕН быть реализован раньше блокируемого шага

**Критерий 4: Группировка по DTO (ОПТИМИЗАЦИЯ)**
- [ ] Несколько TODO требуют изменения ОДНОГО DTO?
- [ ] Если ДА — объединить их в один шаг, чтобы минимизировать запуски build_runner
- [ ] Добавить примечание в плане: "Вместе с TODO #X, #Y"

**Критерий 5: Прогнозирование рисков от TODO**
- [ ] TODO не скрывает потенциальную потерю данных?
- [ ] TODO не создаёт race condition?
- [ ] TODO не приведёт к молчаливому игнорированию ошибок?
- [ ] TODO не создаёт "иллюзию работающей функциональности"?

### Проверка назначения TODO — 5 ВОПРОСОВ

Для каждого TODO задать:
1. [ ] Что именно не реализовано и почему?
2. [ ] Какие компоненты нужны для реализации?

4. [ ] Можно ли реализовать раньше? (если да — перенести)
5. [ ] Можно ли реализовать позже? (если да и безопасно — можно отложить)

## ЧЕК-ЛИСТ: УПРАВЛЕНИЕ ЗАГЛУШКАМИ (STUBS) В КОДЕ

### Типы заглушек и их уровень риска

| Тип | Пример | Уровень риска | Почему |
|-----|--------|--------------|--------|
| **No-op** | `Future<void> save() async { }` | 🔴 КРИТИЧЕСКИЙ | Пользователь думает что операция выполнилась, но данные потеряны |
| **Return empty** | `return [];` | 🟡 СРЕДНИЙ | UI покажет "пусто", но пользователь может не понять что функция не работает |
| **Return hardcoded** | `return SyncStatus.synced;` | 🟡 СРЕДНИЙ | Создаёт иллюзию успеха. Безопасно только если return `pending`/`false` |
| **Hardcoded values** | `pageFormat: 'A4'` | 🟡 СРЕДНИЙ | Теряет настройки пользователя. Безопасно только для новых данных |
| **Partial impl** | Перемещение URL вместо поля | 🟡 СРЕДНИЙ | Работает не как задумано, может сломать ожидания UI |
| **Safe fallback** | `return null;` с проверкой | 🟢 НИЗКИЙ | Caller проверяет null, поведение предсказуемо |

### Алгоритм создания безопасной заглушки

**Шаг 1: Определить тип заглушки**
- [ ] Функция ничего не делает? → **No-op**
- [ ] Функция возвращает пустое значение? → **Return empty**
- [ ] Функция возвращает фиксированное значение? → **Return hardcoded**
- [ ] Параметры захардкожены? → **Hardcoded values**
- [ ] Есть реализация, но неполная? → **Partial impl**

**Шаг 2: Оценить риск (5 вопросов)**
1. [ ] Это функция CRUD (создание/сохранение/обновление/удаление)? → No-op ЗАПРЕЩЁН
2. [ ] Пользователь вызывает эту функцию явно через UI? → No-op ЗАПРЕЩЁН
3. [ ] Заглушка возвращает `true`/`synced`/`success`? → ЗАМЕНИТЬ на `false`/`pending`/`error`
4. [ ] Заглушка теряет настройки пользователя? → ДОБАВИТЬ комментарий о потере данных
5. [ ] Заглушка может быть принята за реальную реализацию? → ДОБАВИТЬ явный TODO

**Шаг 3: Выбрать безопасную заглушку**

| Операция | Безопасная заглушка | Опасная заглушка |
|----------|-------------------|------------------|
| Получить список | `return [];` | `return [fakeItem];` |
| Получить объект | `return null;` | `return defaultObject;` |
| Проверить статус | `return false;` | `return true;` |
| Статус синхронизации | `return SyncStatus.pending;` | `return SyncStatus.synced;` |
| Сохранить данные | `throw UnimplementedError();` | `return;` (молчаливо) |
| Обновить данные | `throw UnimplementedError();` | `return;` (молчаливо) |
| Удалить данные | `throw UnimplementedError();` | `return;` (молчаливо) |

**Шаг 4: Добавить комментарий**
- [ ] Комментарий содержит TODO с шагом плана
- [ ] Комментарий объясняет ПОЧЕМУ это заглушка
- [ ] Комментарий предупреждает о риске (если риск > низкий)

### Алгоритм реализации заглушки (когда дошли до назначенного шага)

1. [ ] Найти ВСЕ заглушки назначенного типа (grep по паттернам)
2. [ ] Проверить, что зависимости для реализации уже существуют
3. [ ] Реализовать полноценную версию
4. [ ] Удалить комментарий заглушки
5. [ ] Проверить flutter analyze
6. [ ] Проверить, что заглушка была заменена полностью (grep ещё раз)

## ЧЕК-ЛИСТ: АНАЛИЗ И ПРОГНОЗИРОВАНИЕ ПЕРЕД ИЗМЕНЕНИЕМ (Правило 17)

### Перед изменением кода
- [ ] Задать вопрос: "К чему приведёт это изменение?"
- [ ] Проверить: не сломает ли это существующие функции?
- [ ] Проверить: не создаст ли это скрытых ошибок на следующих шагах?
- [ ] Проверить: есть ли side effects в конструкторах/методах?
- [ ] Проверить: работает ли это на ОБЕИХ платформах?
- [ ] Если изменяется план — проверить: логичность, целостность, соответствие целям

### После анализа — принятие решения
- [ ] Документировать прогноз (что изменится, какие риски)
- [ ] Принять решение ТОЛЬКО после завершения анализа

## ЧЕК-ЛИСТ: СОХРАНЕНИЕ ФУНКЦИОНАЛЬНОСТИ (Правило 18)

### Перед изменением функции
- [ ] Понять что делала функция ДО рефакторинга (вход, выход, side effects)
- [ ] Найти все точки вызова этой функции
- [ ] Проверить: есть ли зависимости от конкретной реализации

### После изменения функции
- [ ] Сравнить поведение: делает ли функция ТО ЖЕ САМОЕ?
- [ ] Проверить по каждому параметру/полю: результат идентичен?
- [ ] Проверить side effects: изменились ли они?
- [ ] Проверить: не изменилось ли поведение с точки зрения пользователя?
- [ ] Если поведение изменилось — это redesign, требует обсуждения с пользователем

### Критерии эквивалентности рефакторинга
- [ ] Все поля объекта имеют те же значения после изменения
- [ ] Все side effects идентичны (или уменьшены)
- [ ] Производительность не ухудшена (или улучшена)
- [ ] Поведение для пользователя не изменилось
