## Модели данных

### Plant (plant.dart) - Основная модель растения

**Поля:**
- `permanentId` - уникальный ID (UUID)
- `displayId` - отображаемый ID (например: "24-001", "K24-001")
- `latinName` - латинское название
- `status` - статус (sown, growing, in_collection, dead, failed)
- `year` - год
- `customNumber` - порядковый номер
- `category` - категория (sown, purchased)
- `country` - страна происхождения
- `habitat` - местообитание
- `description` - описание
- `synonyms` - синонимы
- `careTips` - советы по уходу
- `floweringPeriod` - период цветения
- `countryFlag` - флаг страны
- `seedsCount` - количество семян
- `germinatedCount` - количество взошедших
- `wateringDates` - даты полива (List<DateTime>)
- `customWateringDates` - кастомные даты полива
- `hasUnreadNotification` - флаг непрочитанного уведомления
- `lastFertilization` - последняя подкормка
- `plannedFertilizationDate` - плановая подкормка
- `lastRepotting` - последняя пересадка
- `plannedTransplantDate` - плановая пересадка
- `germinationHistory` - история всхожести (List<GerminationRecord>)
- `floweringHistory` - история цветения (List<FloweringRecord>)
- `notes` - заметки (List<Note>)
- `userPhotos` - фото пользователя (List<String>)
- `lliflePhotoUrls` - фото из Llifle
- `fieldNumber` - полевой номер
- `seller` - продавец
- `harvestYear` - год сбора семян
- `lastModified` - время последнего изменения
- `gbifPhotoUrls` - фото из GBIF
- `gbifOccurrences` - данные из GBIF (List<GbifOccurrence>)
- `lastGbifUpdate` - последнее обновление GBIF
- `aliveCount` - количество живых сеянцев (для системы партий)
- `isBatch` - является ли витриной-партией
- `childrenIds` - ID сеянцев (для витрины)
- `parentId` - ID витрины (для сеянца)

**Вспомогательные классы:**
- `GerminationRecord` - запись о всхожести (date, germinatedCount, deadCount)
- `FloweringRecord` - запись о цветении (date, event: "bloomed"/"wilted")
- `Note` - заметка (id, title, text, createdAt)
- `GbifOccurrence` - данные из GBIF

**Методы:**
- `copyWith()` - копирование с изменениями
- `toJson()` - сериализация
- `fromJson()` - десериализация
- `generateDisplayId()` - генерация отображаемого ID
- `generateSeedlingDisplayId()` - генерация ID для сеянца
- `statusText` - текст статуса
- `age` - возраст растения
- `lastWateringDate` - дата последнего полива
- `lastWateringText` - текст последнего полива
- `getRecommendedTransplantDate()` - рекомендованная дата пересадки
- `getCurrentAliveCount` - текущее количество живых сеянцев

## Система партий (сеянцы)

### Концепция
- Витрина-партия (isBatch = true) - отображает группу сеянцев
- Сеянцы (parentId != null) - отдельные растения из партии
- childrenIds - список ID сеянцев для витрины
- parentId - ID витрины для сеянца
- aliveCount - количество живых сеянцев (ручной или авторасчёт)

### Генерация ID
- Витрина: "24-001" или "K24-001"
- Сеянцы: "24-001-1", "24-001-2", и т.д.

## Текущая архитектура

### Паттерн архитектуры
- **State Management:** Provider (ChangeNotifier)
- **Хранение данных:** SharedPreferences (JSON)
- **Синхронизация:** Яндекс.Диск (OAuth2)
- **Парсинг:** HTTP запросы к GBIF API и HTML парсинг Llifle

### Поток данных
1. **Загрузка данных:**
   - SharedPreferences → PlantProvider.loadPlants() → _plants
   - Кэширование дат полива → _individualWateringDatesCache

2. **Сохранение данных:**
   - PlantProvider.savePlants() → SharedPreferences
   - CloudStorageProvider.syncData() → Яндекс.Диск

3. **Отображение данных:**
   - PlantProvider → UI (через Consumer/Provider.of)

### Ограничения текущей архитектуры

**1. SharedPreferences как хранилище**
- Ограничение на размер данных (обычно несколько МБ)
- Нет транзакций (если ошибка при сохранении, данные могут быть частично сохранены)
- Нет сложных запросов (нужно загружать все данные в память)
- Нет отношений между данными (все в одной модели Plant)
- Нет индексов (поиск выполняется перебором всех элементов)
- Медленно при большом количестве данных (1000+ растений)

**2. Отсутствие разделения данных**
- Все данные о растении в одной модели Plant
- Нет отдельной модели для вида (SpeciesTemplate)
- Нет отдельной модели для экземпляра (PlantInstance)
- Нет отдельной модели для логов (CareLog)
- Нет отдельной модели для среды (EnvironmentSnapshot)
- Нет отдельной модели для калибровки (CalibrationHistory)

**3. Простая система полива**
- Фиксированные интервалы (не адаптивная)
- Нет учета физиологии растения
- Нет учета среды (температура, влажность, свет)
- Нет учета фазы растения (покой, рост, цветение)
- Нет автокалибровки на основе истории
- Нет индекса готовности к поливу

**4. Отсутствие системы здоровья**
- Только свободные заметки (нет структуры)
- Нет структурированного ввода симптомов
- Нет гипотез причин проблем
- Нет протоколов лечения
- Нет аналитики эффективности методов

**5. Отсутствие машины состояний**
- Нет фаз растения (Dormant, Transition, ActiveGrowth, Flowering, PreDormant)
- Нет автоматического определения фазы
- Нет сезонных триггеров
- Нет правил-блокировок (например, блокировка полива в покое)

**6. Отсутствие планировщика с приоритетами**
- Уведомления без приоритетов
- Нет контекста в уведомлениях (почему поливать)
- Нет группировки уведомлений
- Нет отложенных уведомлений
