# VISION - Финальное состояние приложения

**Дата создания:** 2026-05-09  
**Версия:** 2.0.0 (планируемая)

---

## Миссия

Создать интеллектуального ассистента для коллекционеров кактусов, который:
- Работает офлайн и хранит данные локально
- Адаптируется под конкретные условия пользователя
- Моделирует физиологию растений
- Предлагает уход на основе расчётов, а не фиксированного календаря
- Помогает систематизировать здоровье растений
- Предоставляет профессиональные инструменты для ведения коллекции

---

## Ключевые принципы (из философии дизайна)

### 1. Научная точность + практичность
- Использование научных данных о физиологии растений
- Таксономия соответствует актуальным ревизиям (POWO, Cactaceae databases)
- Практические рекомендации для реального ухода
- Баланс между теорией и опытом
- Интерфейс не должен отпугивать новичков

### 2. Работа в реальных условиях
- Тёплая, влажная, часто без стабильного Wi-Fi теплица
- Приложение должно работать полностью офлайн
- Быстро открываться
- Не слепить экраном
- Быть удобным даже в садовых перчатках
- Минимализм с акцентом на контент
- Темная тема по умолчанию
- Высококонтрастные элементы
- Крупные зоны нажатия
- Шрифты с поддержкой латиницы и диакритики
- Иконки стилизованы под ботанические гравюры, а не под "мультяшные растения"

### 3. Офлайн-First
- Полная функциональность без интернета
- Локальное хранение всех данных
- Синхронизация по требованию
- Никаких обязательных облаков

### 4. Принадлежность данных коллекционеру
- Полный контроль над данными
- Локальное хранение, шифрование
- Экспорт в открытых форматах
- Никаких обязательных подписок за базовый функционал
- Возможность переноса на другой сервер или в настольную СУБД

### 5. Уважение к растению
- Уважение к растению как объекту наблюдения, а не как предмету потребления
- Не пытается заменить опыт, а усиливает его
- Фиксирует, систематизирует, напоминает, сравнивает, сохраняет историю
- Работает в грязи и при ярком солнце
- Не продаёт данные
- Не ломается при смене названия вида
- Растёт вместе с коллекцией: от 10 растений до нескольких тысяч

## Финальная архитектура

### Архитектурный стиль: Offline-First + Event-Driven

**Ключевые принципы архитектуры:**
- **Локальная БД** (SQLite/Realm/WatermelonDB) хранит всё: растения, логи, профили видов, калибровочные коэффициенты
- **Движок расчётов** работает полностью на устройстве. Никаких запросов к серверу для принятия решений
- **Фоновый планировщик** (WorkManager / BackgroundTasks) периодически пересчитывает состояние или реагирует на события (запуск приложения, изменение времени суток, ввод данных)
- **Реактивный UI** подписывается на изменения состояния растений и обновляет карточки/уведомления без перезагрузки

**Слои архитектуры:**
1. **Data Layer**: локальная БД с таблицами растений, видов, логов, среды, калибровки
2. **Logic Layer**:
   - PhaseStateMachine (переходы по порогам)
   - ReadinessCalculator (детерминированная формула с весами)
   - CalibrationEngine (EMA-подстройка на основе истории)
3. **Scheduler Layer**: очередь задач, приоритизация, доставка уведомлений, quiet hours
4. **UI Layer**: реактивные подписки на изменения состояния, рендер статуса + контекста, формы ввода логов
5. **Sync/Backup Layer** (опционально): шифрованный экспорт/импорт JSON, облачная синхронизация по требованию

**Ядро:**

## Модели данных (5 сущностей)

### 1. SpeciesTemplate (Таксономический профиль вида)
**Назначение:** Хранит базовую информацию о виде как таксономической единице

**Поля:**
- `id` - уникальный ID
- `taxon_id` - таксономический ID
- `scientificName` - научное название
- `synonyms` - синонимы (история изменений, при обновлении базы старые названия не исчезают)
- `family` - семейство
- `genus` - род
- `nativeRange` - естественный ареал
- `habitat` - местообитание
- `growthForm` - форма роста (столбик, шаровидный, кустарниковый)
- `sizeMature` - размер взрослого растения
- `coldHardiness` - холодостойкость (минимальная температура)
- `droughtTolerance` - засухоустойчивость
- `lightRequirements` - требования к свету
- `soilType` - тип почвы
- `wateringNeeds` - потребность в поливе (базовый профиль)
- `activity_season` - тип цикла (летнеактивный, зимнеактивный, переходный)
- `min_active_temp` - критическая температура для выхода из покоя
- `dormancy_multiplier` - множитель покоя
- `base_dry_days` - базовое количество дней высыхания
- `substrate_tolerance` - толерантность к субстрату
- `floweringSeason` - сезон цветения
- `dormancyPeriod` - период покоя
- `propagationMethods` - методы размножения
- `careGuidelines` - базовые рекомендации по уходу
- `gbifData` - данные из GBIF
- `llifleData` - данные из Llifle
- `createdAt` - дата создания
- `updatedAt` - дата обновления

**Связи:**
- Один SpeciesTemplate → много PlantInstance

---

### 2. PlantInstance (Конкретное растение)
**Назначение:** Хранит информацию о конкретном экземпляре растения в коллекции

**Поля:**
- `id` - уникальный ID
- `species_id` - ссылка на SpeciesTemplate
- `displayId` - отображаемый ID (например: "24-001", "K24-001")
- `acquisitionDate` - дата приобретения
- `acquisitionSource` - источник приобретения (семена, покупка, обмен)
- `acquisitionDetails` - детали приобретения (seller, fieldNumber, harvestYear)
- `pot_volume` - объём горшка (мл)
- `pot_material` - материал горшка (пластик, керамика, терракота)
- `substrate_porosity` - пористость субстрата (% минералов, влагоёмкость, pH, дренаж)
- `currentPhase` - текущая фаза (Dormant, Transition, ActiveGrowth, Flowering, PreDormant)
- `phaseHistory` - история смены фаз
- `locationId` - ссылка на локацию (место в теплице)
- `healthStatus` - статус здоровья (healthy, issues, critical)
- `lastHealthCheck` - последняя проверка здоровья
- `customNotes` - пользовательские заметки
- `userPhotos` - фото пользователя
- `referencePhotos` - референсные фото (для сравнения)
- `qrCode` - QR-код
- `isBatch` - является ли витриной-партией
- `parentInstanceId` - родительский экземпляр (для сеянцев)
- `childInstanceIds` - дочерние экземпляры (для витрины)
- `aliveCount` - количество живых сеянцев
- `last_care_date` - дата последнего ухода
- `readiness_index` - текущий индекс готовности (0-100)
- `calibration_offset` - калибровочный коэффициент
- `createdAt` - дата создания
- `updatedAt` - дата обновления

**Связи:**
- Один PlantInstance → один SpeciesTemplate
- Один PlantInstance → много CareLog
- Один PlantInstance → много EnvironmentSnapshot
- Один PlantInstance → много CalibrationHistory
- Один PlantInstance → одна Location

### 3. CareLog (Журнал ухода)
**Назначение:** Хранит историю всех действий ухода за растением

**Поля:**
- `id` - уникальный ID
- `plant_id` - ссылка на PlantInstance
- `timestamp` - дата и время

- `environmentSnapshotId` - ссылка на EnvironmentSnapshot (если есть)
- `readinessIndex` - индекс готовности на момент ухода
- `phase` - фаза растения на момент ухода
- `photos` - фото до/после
- `userInitiated` - инициировано пользователем или системой
- `createdAt` - дата создания

**Связи:**
- Один CareLog → один PlantInstance
- Один CareLog → один EnvironmentSnapshot (опционально)

### 4. EnvironmentSnapshot (Состояние среды)
**Назначение:** Хранит состояние окружающей среды в конкретный момент времени

**Поля:**
- `id` - уникальный ID
- `location_id` - ссылка на локацию
- `timestamp` - дата и время
- `temp_day` - температура днём (°C)
- `temp_night` - температура ночью (°C)
- `light_hours` - световой день (часы)
- `light_level` - уровень освещенности (lux)
- `humidity` - влажность воздуха (%)
- `soil_moisture` - влажность почвы (%)
- `ventilation` - вентиляция (наличие/отсутствие)
- `season` - сезон
- `source` - источник данных (manual, sensor, automatic)
- `sensor_id` - ID датчика (если применимо)
- `notes` - заметки
- `createdAt` - дата создания

**Связи:**
- Один EnvironmentSnapshot → одна Location (опционально)
- Один EnvironmentSnapshot → много CareLog

### 5. CalibrationHistory (История адаптации)
**Назначение:** Хранит историю адаптации системы под конкретные условия

**Поля:**
- `id` - уникальный ID
- `plant_id` - ссылка на PlantInstance
- `timestamp` - дата и время
- `predicted_dry_days` - предсказанные дни высыхания (по модели)

- `environmentContext` - контекст среды (ссылка на EnvironmentSnapshot)

- `createdAt` - дата создания

**Связи:**
- Один CalibrationHistory → один PlantInstance
- Один CalibrationHistory → один EnvironmentSnapshot (опционально)

## Адаптивный график ухода: 4 слоя данных

Система опирается на 4 слоя данных, которые постоянно обновляются:

### Слой 1: Таксономическо-биологический
**Что учитывается:**
- Природный ареал
- Тип цикла (летне-/зимнеактивный, переходный)
- Критические температуры для выхода/входа в покой
- Чувствительность к влаге
- Требования к свету
- Скорость роста в природе

**Как поступает:**
- Загружается из офлайн-базы при создании карточки
- Позволяет ручную корректировку, если растение культиварное или привитое

### Слой 2: Физико-средовой
**Что учитывается:**
- Материал и объём горшка
- Состав субстрата (% минералов, влагоёмкость, pH)
- Локация (стеллаж, окно, теплица, улица)
- Реальная t° д/н
- Освещённость (лк/ч)
- Влажность воздуха
- Вентиляция

**Как поступает:**
- Вводится при добавлении или импортируется из датчиков (Bluetooth/Wi-Fi)
- Без датчиков работает на ручном вводе + климатической модели региона

### Слой 3: Физиологический
**Что учитывается:**
- Дата последнего полива
- Текущая влажность субстрата (визуально/щуп/датчик)
- Фаза (покой, пробуждение, бутонизация, рост, плодоношение)
- Недавние операции (пересадка, удобрение, обработка)
- Признаки стресса или роста

**Как поступает:**
- Вводится пользователем или фиксируется автоматически через лог действий

### Слой 4: Пользовательский
**Что учитывается:**
- Стиль ухода (консервативный / умеренный / экспериментальный)
- Допустимые отклонения
- Ручные коррекции
- История эффективности методов

**Как поступает:**
- Настраивается в профиле коллекционера
- Система запоминает ваши привычки и калибруется под них

## Машина состояний растения (Phase State Machine)

### Фазы растения

**1. Dormant (Покой)**
- Минимальный полив
- Низкая температура
- Нет роста
- Длительность: зависит от вида и сезона

**2. Transition (Переход)**
- Подготовка к активной фазе
- Увеличение полива
- Начало роста
- Длительность: 1-2 недели

**3. ActiveGrowth (Активный рост)**
- Максимальный полив
- Оптимальная температура
- Активный рост
- Длительность: зависит от сезона

**4. Flowering (Цветение)**
- Специфический полив
- Может быть меньше чем в ActiveGrowth
- Цветение
- Длительность: 2-6 недель

**5. PreDormant (Подготовка к покою)**
- Постепенное сокращение полива
- Снижение температуры
- Подготовка к зимовке
- Длительность: 2-4 недели

### Правила переходов
- Dormant → Transition: сезонный триггер (весна)
- Transition → ActiveGrowth: признаки роста
- ActiveGrowth → Flowering: сезонный триггер + физиологическая готовность
- Flowering → ActiveGrowth: завершение цветения
- ActiveGrowth → PreDormant: сезонный триггер (осень)
- PreDormant → Dormant: завершение подготовки
- Любая фаза → Dormant: стресс/болезнь

## Readiness Engine (Движок готовности к уходу)

### Индекс готовности к поливу (0-100%)

Это ядро системы. Не календарь, а детерминированная функция, возвращающая число 0-100%.

**Формула концептуально:**
```
ReadinessIndex = BaseIntervalProgress × TemperatureFactor × SubstrateFactor × LightFactor × PhaseMultiplier × CalibrationOffset
```

**Где:**

**1. BaseIntervalProgress (Прогресс базового интервала)**
- Сколько дней прошло с последнего полива / сколько дней в среднем сохнет субстрат этого вида
- Диапазон: 0-100%

**2. TemperatureFactor (Температурный множитель)**
- Модель испарения. Выше t° → быстрее сохнет
- Учитывает разницу день/ночь
- Оптимальная = 1.0, отклонение = 0.5-1.5

**3. SubstrateFactor (Множитель субстрата)**
- 80% пемзы сохнет в 2 раза быстрее, чем чистый торф
- Задаётся коэффициентом влагоёмкости
- Диапазон: 0.5-2.0

**4. LightFactor (Световой множитель)**
- Фотосинтетически активный свет ускоряет метаболизм и транспирацию
- Оптимальный = 1.0, отклонение = 0.8-1.2

**5. PhaseMultiplier (Множитель фазы)**
- 0.0 в покое
- 0.3 в переходе
- 1.0 в активном росте
- 1.2 при бутонизации

**6. CalibrationOffset (Калибровочный сдвиг)**
- Динамический коэффициент, который система подтягивает на основе истории
- Диапазон: 0.7-1.3

### Реализация в коде

- Функция вызывается при запуске приложения, при вводе новых данных или по фоновому таймеру
- Возвращает объект: `{ index: 78, status: "check_soon", reason_chain: ["temp +24°C", "substrate 70% mineral", "phase active", "calibrated -2 days"] }`
- UI рендерит статус, а не просто число

### Интерпретация индекса
- **0-20:** Не поливать (растение в покое или проблемы)
- **21-40:** Поливать не скоро (низкая готовность)
- **41-60:** Поливать через неделю (умеренная готовность)
- **61-80:** Поливать через 3-5 дней (хорошая готовность)
- **81-100:** Поливать сейчас или завтра (высокая готовность)

## Правила-триггеры и блокировки

Система использует детерминированные правила для принятия решений:

### Правила блокировки полива

**1. Температурная блокировка для летнеактивных видов**
- **Условие:** IF t° ночи < +10°C OR t° дня < +15°C AND вид летнеактивный
- **Действие:** Статус "Покой". Полив заблокирован
- **Причина:** Риск корневой гнили при низкой температуре

**2. Риск перегрева корней**
- **Условие:** IF t° дня > +32°C
- **Действие:** Предупреждение "Риск сваривания корней"
- **Рекомендация:** Полив в вечернее время или метод погружения

**3. Адаптация после пересадки**
- **Условие:** IF пересадка < 14 дней назад
- **Действие:** Статус "Адаптация"
- **Рекомендация:** Полив только лёгкий, если есть признаки укоренения

### Правила ускорения цикла

**1. Обнаружение активного роста**
- **Условие:** IF обнаружен новый прирост/бутон
- **Действие:** Система ускоряет цикл проверок
- **Рекомендация:** Разрешает подкормку

**2. Сезонные окна**
- **Зимнеактивные виды:** Ноябрь--февраль = активная фаза
- **Летнеактивные виды:** Май--сентябрь = активная фаза
- **Переходные виды:** Адаптивный график на основе температуры

### Безопасные overrides

Если пользователь решает нарушить рекомендацию (например, дать микрополивы зимой):
- Система не блокирует
- Помечает запись как "Эксперимент/Коррекция"
- Предупреждает о рисках
- Просит указать цель
- Данные не удаляются, а сохраняются для обучения

## Механизм адаптации и калибровки (Feedback Loop)

### EMA (Exponential Moving Average)

**Принцип:** Экспоненциальное скользящее среднее для плавной адаптации под условия пользователя

**Формула:**
```
EMA_new = α × CurrentValue + (1 - α) × EMA_old
```

Где α (alpha) - коэффициент сглаживания (обычно 0.1-0.3)

### Процесс калибровки

**1. Сбор данных**
- Пользователь поливает растение
- Система фиксирует EnvironmentSnapshot
- Система записывает CareLog

**2. Анализ**
- Сравнение рекомендованной даты с фактической
- Анализ состояния растения после полива
- Анализ среды

**3. Корректировка**
- Если пользователь поливает раньше → уменьшить интервал
- Если пользователь поливает позже → увеличить интервал
- Запись в CalibrationHistory

**4. Применение**
- Обновление параметров в SpeciesTemplate или PlantInstance
- Пересчет ReadinessIndex

**5. Валидация**
- Проверка результатов следующего полива
- Корректировка confidence

### Ограничения
- Минимальный интервал: 3 дня
- Максимальный интервал: 60 дней
- Максимальная корректировка за раз: ±20%
- Минимальный confidence: 0.3

## Система здоровья (подробное описание)

Это не "диагност по фото", а структурированная система ведения медицинских карт растений. Её цель — не угадать диагноз, а помочь коллекционеру зафиксировать наблюдение, сузить круг возможных причин, применить метод, отследить результат и сохранить опыт для будущего.

### Структура вкладки "Здоровье" (6 блоков)

#### 1️⃣ Сводка статуса (верхний баннер)

Отображается всегда. Даёт мгновенное понимание состояния.

**Элементы:**
- **Индикатор здоровья:** Цветовая метка 🟢 Норма / 🟡 Наблюдение / 🟠 Проблема / 🔴 Критично
- **Активная проблема:** Последний незакрытый симптом с датой (например: "Хлороз шейки (с 12.03)")
- **Риск-факторы:** Автоматически подсвеченные условия, повышающие уязвимость (например: "Высокая влажность воздуха + низкая вентиляция")
- **Быстрые действия:** Кнопки "Отметить улучшение", "Добавить фото", "Запросить совет"

#### 2️⃣ ➕ Добавить симптом / событие

**Не свободный текст, а структурированный ввод через таксономию симптомов.**

**Шаг 1: Выбор категории симптома**
- Визуальные изменения (изменение цвета, формы, повреждения тканей, налёты/образования)
- Поведенческие сигналы (отсутствие роста, сброс бутонов, замедление реакции)
- Контекстные события (после пересадки, после обработки, после транспортировки)

**Шаг 2: Уточнение локализации и масштаба**
- **Где:** основание стебля / ареолы / точка роста / корни / вся поверхность
- **Сколько:** единично / несколько участков / >50% растения
- **Динамика:** появилось внезапно / нарастает 3-5 дней / стабильно неделю

**Шаг 3: Привязка контекста (автоматическая + ручная)**
- Система подтягивает: последние 7 дней (температура, влажность, полив, подкормка), фазу растения, историю похожих симптомов
- Пользователь добавляет: фото (обязательно с масштабом — линейка/монета), голосовую заметку, теги (#подозрение_на_гниль, #после_пересадки, #карантин)

**Шаг 4: Первоначальная оценка риска (системная)**
- 🟢 Низкий: косметический дефект, нет прогрессии
- 🟡 Средний: требует наблюдения в 48-72 ч
- 🟠 Высокий: риск распространения, рекомендована изоляция
- 🔴 Критичный: угроза жизни растения, немедленное вмешательство

#### 3️⃣ 📋 История проблем (хронологическая лента)

Каждая запись — это "эпизод болезни" со структурой:
- Дата и название симптома
- Статус (🔴 Активно / 🟡 Наблюдение / 🟢 Решено)
- Локализация
- Контекст (после полива, t° ночи)
- Действия (осмотр, фото, изоляция, обработка)
- Эффективность (оценивается пользователем)
- Связанные записи (фото до/после, лог полива)

**Фичи истории:**
- Фильтр: "Только активные", "Только решённые", "По типу симптома"
- Сравнение эпизодов: "Показать все случаи хлороза у этого вида"
- Экспорт эпизода в PDF для консультации с экспертом

#### 4️⃣ 🔍 Контекстная база знаний (не "диагноз", а "гипотезы + проверки")

После ввода симптома система не говорит "это гниль", а показывает:

**Пример гипотез:**
1️⃣ Переувлажнение в прохладных условиях
- Почему вероятно: совпадает с поливом 10.03 + t° ночи < +15°C
- Что проверить: сухость субстрата на глубине 3-5 см, запах от грунта, состояние корней
- Рекомендуемые действия: прекратить полив до полного высыхания, усилить вентиляцию
- Источники: [Cactaceae Care Manual, p.87] [Ваш лог #142]

2️⃣ Механическое повреждение при пересадке + вторичная инфекция
- Почему возможно: пересадка 05.03, симптом появился через неделю
- Что проверить: наличие раневой поверхности, налёт мицелия
- Действия: обработка среза серой/фунгицидом, сухое содержание

**Важно:** каждая гипотеза сопровождается:
- Уровнем соответствия вашим данным (высоко/средне/низко)
- Конкретными, выполнимыми проверками ("пощупайте", "понюхайте", "посмотрите")
- Приоритетом действий (что сделать в первую очередь)
- Ссылками на ваши прошлые успешные кейсы

#### 5️⃣ 💊 Протоколы лечения и лог вмешательств

Когда вы выбираете действие, система предлагает структурированный протокол:

**Пример протокола:**
📋 Протокол: Препарат Х (системный, на основе тебуконазола)
- Дозировка: 0.5 мл/л воды (для опрыскивания шейки)
- Метод: точечная аппликация ватной палочкой, не поливать!
- Частота: однократно, повтор через 7 дней только при прогрессии
- Меры безопасности: перчатки, проветривание, не смешивать с удобрениями
- Ожидаемый эффект: остановка потемнения в течение 48-72 ч
- Маркеры ухудшения: распространение пятна >1 см/день, запах гнили

После применения:
- В лог автоматически добавляется запись с датой, препаратом, дозировкой
- Создаётся задача на контрольный осмотр
- Система блокирует противоречащие действия (например, полив в ближайшие 24-48 ч)

#### 6️⃣ 📈 Аналитика эффективности (для продвинутых)

Доступна при наличии ≥3 записей по одному типу симптома.

**Пример анализа:**
📊 Анализ эффективности методов при "хлороз шейки":
- Просушка + вентиляция: 4 случая, 100% успешность, средний срок 5.2 дня
- Фунгицид Х: 2 случая, 50% успешность, средний срок 8.0 дней
- Экстренная пересадка: 1 случай, 100% успешность, средний срок 14 дней

💡 Вывод системы: Для ваших условий (теплая теплица, минеральный субстрат) наиболее эффективна комбинация "просушка + усиление вентиляции". Фунгициды дают результат только при раннем применении.

### Кросс-реферсы с другими вкладками

- При вводе симптома "этиоляция" система автоматически проверяет вкладку "Условия" → "Освещённость" и подсвечивает: "Средняя освещённость за последние 14 дней: 3 200 лк. Для этого вида рекомендовано >8 000 лк"
- При симптоме "гниль" проверяется "График ухода" → "Последний полив" и "Субстрат" → "Влагоёмкость"

### Экспорт для консультации

- Кнопка "Поделиться эпизодом" генерирует анонимизированный PDF: фото, симптомы, контекст, предпринятые действия — без личных данных, но с полной ботанической информацией
- Идеально для отправки эксперту или в сообщество

### Принцип работы

**1. Ввод симптома**
- Пользователь выбирает категорию
- Уточняет локализацию
- Добавляет контекст
- Оценивает риск
- Прикладывает фото

**2. Генерация гипотез**
- Система анализирует симптом
- Предлагает 2-3 гипотезы причин
- Для каждой гипотезы предлагает проверки

**3. Проверка гипотез**
- Пользователь выполняет проверки
- Отмечает результаты
- Система сужает круг причин

**4. Выбор протокола**
- На основе подтвержденной гипотезы
- Пользователь выбирает протокол лечения
- Или создает свой

**5. Лечение**
- Пошаговое выполнение
- Фиксация прогресса
- Фото до/после

**6. Анализ**
- Оценка эффективности лечения
- Запись в историю
- Обучение системы

### Примеры

**Корневая гниль:**
- Симптом: размягчение основания стебля
- Гипотеза 1: Корневая гниль от переувлажнения
- Проверка 1: Осмотр корней
- Проверка 2: Проверка влажности почвы
- Протокол: Извлечение, очистка, обработка фунгицидом, пересадка в сухую почву

**Паутинный клещ:**
- Симптом: Паутина на стебле
- Гипотеза 1: Паутинный клещ
- Проверка 1: Осмотр под лупой
- Проверка 2: Проверка соседних растений
- Протокол: Промывка, акарицид, изоляция

**Солнечный ожог:**
- Симптом: Коричневые пятна
- Гипотеза 1: Солнечный ожог
- Проверка 1: Проверка освещенности
- Проверка 2: Анализ расположения
- Протокол: Перестановка в тень, мониторинг

## Типичный сценарий использования (Workflow)

### 1. Добавление нового растения

**Действия пользователя:**
- Сканирую QR с этикетки поставщика или ввожу вручную
- Приложение предлагает таксономию из базы, показывает синонимы и статус CITES
- Загружаю первое фото, указываю locality data, collector's number, дату приобретения, стоимость
- Выбираю локацию (стеллаж A3, полка 2), задаю начальные условия (грунт: 70% пемза + 30% торф, горшок 8 см, освещение: 12 ч/день)

**Действия системы:**
- Автоматически создаёт адаптивный график ухода
- Привязывает к календарю
- Рассчитывает базовый индекс готовности

### 2. Ежедневное/еженедельное обслуживание

**Действия пользователя:**
- Получаю уведомление: Echinocereus enneacanthus #EE-089: проверить субстрат на сухость
- Захожу в теплицу, ощупываю грунт, отмечаю "влажно" → график сдвигается
- Делаю фото, добавляю заметку: "появился бутон, температура ночью +14°C"

**Действия системы:**
- Автоматически помечает растение как "в активной фазе"
- Корректирует частоту проверок
- Обновляет индекс готовности

### 3. Работа с проблемой

**Действия пользователя:**
- Замечаю желтизну у основания
- Открываю карточку → вкладка "Здоровье" → выбираю симптом "хлороз/загнивание шейки"
- Добавляю фото, тег #после_полива

**Действия системы:**
- Показывает историю подобных случаев у этого вида/в моих условиях
- Предлагает проверить дренаж, снизить полив, обработать фунгицидом
- Присваивает статус "🟠 Высокий риск", блокирует рекомендации по поливу на 7 дней

**Действия пользователя:**
- Фиксирую обработку, делаю фото через 7 дней

**Действия системы:**
- Сохраняет эффективность метода
- Закрывает эпизод с меткой "✅ Решено"
- Добавляет данные в аналитику

### 4. Сезонная аналитика

**Действия пользователя:**
- В конце сезона открываю "Отчёты" → смотрю статистику цветения, успех прививок, расход субстратов
- Экспортирую PDF-каталог для страховой оценки или научного архива
- Обновляю таксономию (если вышла новая ревизия)

**Действия системы:**
- Старые записи не ломаются, только добавляются примечания
- Анализирует эффективность методов лечения
- Предлагает правила для будущих сезонов

## Итоговое видение

**Приложение My Cactus 2.0** - это:
- Интеллектуальный ассистент для коллекционеров
- Офлайн-First система с полной функциональностью
- Адаптивная система ухода на основе физиологии растений
- Структурированная система здоровья с гипотезами и протоколами
- Профессиональные инструменты для ведения коллекции
- Полный контроль над данными
- Кроссплатформенность (Android, Windows, iOS, Linux, macOS, Web)

- От фиксированного календаря → к адаптивной системе
- От простых заметок → к структурированной системе здоровья
- От одной модели → к 5 связанным моделям
- От SharedPreferences → к локальной БД
- От статических рекомендаций → к динамическим с автокалибровкой

**Цель:** Стать незаменимым инструментом для серьезных коллекционеров кактусов, сочетая научную точность с практической пользой.
